tdd튜토리얼을 따라하며 중요하다고 느낀점 몇가지를 기록용으로 남겨둔다.

### repl처럼 항상 정해진 루프를 거치는게 중요하다.

번거로워 보이더라도 우선 테스트코드를 작성해서 원하는 작동방식을 정의하고
테스트를 통과하게 코드를 작성한 다음 리팩토링을 하는 과정을 거치면 

개발 단위를 분리하여 작업하는데 도움이 되고 개발 만족도 측면에서도 좋다.

성급한 추상화로 인하여 발생하는 문제도 방지할 수 있다.

### 기능단위로 테스트를 나누는게 중요하다.

상세 구현내용을 테스트하기보다 항상 기능 단위로 나눠야
테스트 코드를 비교적 간결하게 유지할 수 있다.

### 테스트 적정 케이스 수

> Write tests until fear is transformed into boredom.

### 테스트 코드가 통과되면 일단 커밋하자.

커밋을 미리 해놓고 리팩토링이 만족스러우면 --amend로 커밋을 수정하는 것도 좋다.

### 관심사의 분리가 잘 적용되었는지 확인하자

원본 파일에 너무 많은 기능이 모여있다면 테스트 코드또한 [복잡해 질 수 밖에 없다](http://quii.gitbook.io/learn-go-with-tests/meta/anti-patterns#excessive-setup-too-many-test-doubles-etc).
제대로 분리되었는지 확인하기 위해서는 해당 파일이 하나의 기능만을 담당하는지 확인해야 한다.
