쓰레드와 비슷하게 db도 여러 client가 공유하는 자원이 존재하기 때문에 
동시성 제어를 위해서 락이라는 개념이 존재하고 단일 op에 대해서는 
db엔진이 자체적으로 락을 처리하지만 트랜잭션 처리가 필요한 경우
직접 락을 걸어야 하는 경우가 생긴다.

흔한 상황으로 유저 포인트를 관리할때 포인트 변경 내용자체를 기록하는 작업과
유저 포인트 값을 update하는 작업을 트랜잭션으로 묶더라도(원자성) 동시에 한 유저에 대한 업데이트를 진행하면 
lost update가 발생할 수 있다.

트랜잭션 내부에서 유저를 업데이트하는 부분에 락을 걸어서 해결할 수도 있지만
직접적인 락은 추후 다른 트랜잭션을 추가하면서 발생할 수 있는 side effect를 줄이기 위해서 최대한 지양하는게 좋은 것 같다.

락을 걸지않더라고 db자체적으로 이러한 연산에 대해서는 원자성을 보장하는 경우가 많아서
[문서](https://www.mongodb.com/ko-kr/docs/manual/core/write-operations-atomicity/)를 찾아보고 적용하면 좋을 것 같다.

예전에는 굳이 update 함수가 있는데 왜 orm에서 increment 함수를 왜 별도로 만들었나 싶었는데
이러한 문제 때문에 만든 것 같다.
